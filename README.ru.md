# 📨 Hormigas Messenger

> Чат **мастер ↔ клиент** в реальном времени для маркетплейса услуг Hormigas — реактивный
> WebSocket-сервис с гарантированной доставкой на **Java 25 + Quarkus + Mutiny** и чистым
> **гексагональным** ядром.
>
> 🌍 Другие языки: [English](./README.md)

Работает рядом с остальными сервисами Hormigas (MasterProfile `:8080`, ClientProfile `:8081`,
Order/TaskManager `:8082`) и позволяет двум участникам задачи — **мастеру** и **клиенту** —
координироваться: обсудить объём, согласовать условия, обмениваться файлами и получать системные
уведомления.

> **Об источнике истины.** Этот README сгенерирован из утверждённого **концепта** и
> **функциональных требований (FR)**, которые хранятся в архитектурном цифровом двойнике
> (`knowledge/concepts/messenger-{concept,use-cases,functional-requirements}.md`). Документы
> двойника остаются авторитетными; README — это публичное представление.

## 🚀 Что это

Мессенджер на **реактивном конвейере**: каждое сообщение проходит через асинхронные стадии
(валидация → сохранение → ACK → доставка → финализация), которые резолвер собирает динамически по
типу сообщения. **Чистый домен (core)** изолирован от инфраструктуры через **порты/адаптеры**, так
что PostgreSQL, Redis и WebSocket-транспорт можно заменить, не трогая бизнес-логику.

### Ключевые технологии

| Компонент | Технология | Назначение |
|-----------|-----------|------------|
| ☕ **Java 25** | Язык | Records, pattern matching, производительность |
| ⚡ **Quarkus 3.32** | Фреймворк | Быстрый старт, низкое потребление, `websockets-next` |
| 🔁 **Mutiny** | Реактивная библиотека | Неблокирующие конвейеры `Uni` |
| 🧠 **Гексагональная архитектура** | Паттерн | Домен изолирован от инфраструктуры |
| 🧱 **PostgreSQL** (реактивный PG-клиент + Flyway) | Надёжное хранилище | Outbox + история сообщений + разговоры |
| ⚙️ **Redis** | In-memory состояние | Presence, идемпотентность, watermark'и Tetris |
| 🔐 **Ory** (Kratos + Oathkeeper) | Идентичность | Заголовки идентичности от прокси (без JWT в приложении) |
| 🌐 **API Gateway** | Граница | TLS, инъекция идентичности, маршрутизация WS, будущий шардинг |

## 🧭 Модель разговора

- **Идентичность** — разговор определяется **парой `(clientId, masterId)`** (две Ory-идентичности).
  **Только 1:1; групповых чатов нет.**
- **Универсальное идемпотентное создание** — одна базовая операция *создания чата*, вызываемая
  взаимозаменяемыми входящими адаптерами. Та же логическая пара → возвращается существующий чат,
  дубликат не создаётся.
  - ✅ **REST**-адаптер (`POST /api/chats`) и вызовы **admin/service** — реализованы.
  - 🟡 **Event-адаптер** ("мастер заинтересовался заказом", Kafka `order.events`) — **отложен**
    (см. [Дорожную карту](#-дорожная-карта-и-отложенное)). Kafka-слушатель — это просто ещё один
    адаптер поверх той же базовой операции.
- **Без привязки к заказу** — **один разговор на пару, переиспользуется между заказами.** Мессенджер
  никогда не моделирует/не ветвится/не группирует по заказу; `orderId` передаётся только как
  **непрозрачные метаданные**. Группировка по заказу — забота **фронтенда**.
- **Мягкое удаление (для каждого участника)** — любая сторона может скрыть чат из своего
  представления; из системы он **никогда не удаляется**, и собеседника это не затрагивает.
- **Чёрный список/блокировка (для каждого участника)** — любая сторона может заблокировать другую;
  пока действует блокировка, отправка между ними отклоняется. Разблокировка восстанавливает обмен.

> **Принцип dual-driven.** Каждая событийная операция (создание чата, заморозка) — это **одна базовая
> операция (use case)**, доступная через **два входящих адаптера — REST и event-консьюмер**. Триггеры
> взаимозаменяемы; механизмом является сам use case. (Сейчас REST работает, event-консьюмер отложен.)

## 💬 Виды сообщений и стратегии обработки

Каждый вид сообщения отображается на одну из четырёх **стратегий обработки** (ортогональные
свойства, резолвятся по `MessageType`):

| Стратегия | Надёжно сохраняется | Ретраи до ACK | Идемпотентна | Хранится в History | Для чего |
|----------|:---:|:---:|:---:|:---:|----------|
| **A — persistent** | да | да | да | да (до TTL; frozen → свой более долгий TTL) | сообщения чата |
| **B — fire-and-forget** | нет | нет | нет | нет | печатает, presence |
| **C — retry-then-purge** | временно (outbox) | да | да | нет | обязательные системные уведомления |
| **S — signaling** | нет (transient) | да | да | нет | установка **WebRTC**-звонка |

> Реализованы сегодня: **A** (чат) и **S** (маршруты сигнализации). **B** — presence. **C**
> (retry-then-purge для системных уведомлений) — **специфицирована, но пока не отдельный pipeline**:
> использует кэширующий путь S, пока не заведена своя семантика хранения.

### Значения `MessageType` на проводе

```
CHAT_IN     CHAT_OUT     CHAT_ACK        # постоянный чат (стратегия A) + ACK доставки
READ_IN     READ_OUT                     # квитанции прочтения (стратегия B): читатель → сервер, сервер → отправитель
SIGNAL_IN   SIGNAL_OUT   SIGNAL_ACK      # сигнализация WebRTC (стратегия S)
PRESENT_INIT  PRESENT_JOIN  PRESENT_LEAVE   # presence (стратегия B)
SERVICE_OUT                              # техническое сообщение сервер→клиент
```

Канонический `messageId` (**ULID** — монотонный по времени) присваивает сервер на входе; собственный
id клиента передаётся как `correlationId` для сопоставления ACK и дедупликации на фронтенде.

## 🔄 Состояния и жизненный цикл сообщения

Сохраняемая машина состояний: **`SENT → DELIVERED → READ`**.

- **SENT** — сообщение надёжно записано (History + Outbox в одной транзакции) и подтверждено
  отправителю.
- **DELIVERED** — устанавливается, когда **получатель присылает ACK о доставке** (`CHAT_ACK`,
  `correlationId` = id доставленного сообщения). Push по WS — best-effort, поэтому DELIVERED
  подтверждается ACK'ом, а не предполагается по факту push. Тот же ACK двигает watermark GC.
- **READ** — получатель шлёт `READ_IN` по WS (fire-and-forget); сервер сохраняет `READ` и пушит
  `READ_OUT` отправителю (тот же realtime-канал, что и DELIVERED). `POST /api/chats/{id}/read` —
  резервный путь для догона/bulk; оба идут через один core `markRead`.

### Счастливый путь для persistent (стратегия A)

1. Клиент → WS `CHAT_IN` `{conversationId, messageId, payload, metadata{orderId}}`.
2. **Валидация** — членство (отправитель в паре, не в чёрном списке), формат, таймзона.
3. **Атомарное сохранение** — History + Outbox в одной транзакции.
4. **ACK отправителю** (`CHAT_ACK`, статус `SENT`).
5. **Доставка** — если получатель онлайн, отправляем `CHAT_OUT` (→ `DELIVERED`); иначе держим в Outbox.
6. **Подтверждение прочтения** → статус `READ`.
7. **GC** — удаляем строки Outbox ≤ безопасного watermark'а; History сохраняется по TTL.

## 🧊 Неизменяемость, заморозка и хранение

- **Неизменяемость** — сообщения записываются один раз; **операции редактирования нет.**
- **Условное удаление** — участник может удалить сообщение **только пока оно НЕ заморожено**;
  замороженное → `409`.
- **Заморозка на уровне сообщения, по `orderId`** — **заморозки на уровне чата нет**. Когда стороны
  достигают договорённости по заказу, замораживаются сообщения *этого заказа* (необратимо), и они
  становятся неизменяемым доказательством. Сообщения другого заказа в **том же** чате остаются
  удаляемыми. `POST /api/chats/{id}/freeze {"orderId": "..."}`.
- **Классы хранения** — обычная история использует TTL по виду; **замороженные** сообщения имеют
  **свой, более долгий** TTL (отдельный класс, не исключение). Репозиторий учитывает оба
  (`deleteOlderThan` исключает frozen; `deleteFrozenOlderThan` — для класса frozen). *(Привязка
  чисток хранения к планировщику — оставшаяся задача.)*

## 🧾 Базовые гарантии

1. **Постоянный чат — не теряется.** Сохраняется надёжно до любого ACK.
2. **Офлайн-доставка.** Держится в Outbox и доставляется при следующем подключении получателя.
3. **At-least-once + best-effort идемпотентность.** Сервер старается не пересылать; exactly-once
   **не** гарантируется — **фронтенд дедуплицирует по `messageId`**.
4. **Подтверждения прочтения** сохраняются (`SENT → DELIVERED → READ`).
5. **Порядок внутри разговора** — *цель; пока не реализовано* (см. Дорожную карту).
6. **Одна активная сессия на пользователя.** Новое WS-подключение клиента вытесняет предыдущее
   (чистый takeover). Мультидевайс в v1 вне области (ACK/watermark/ordering — на уровне клиента).

### 📐 Контракт доставки и инварианты

Система **без потерь, at-least-once**, а не «гарантированный live-push». Точно:

- **`message_history` — единственный источник истины** и не теряет сообщений в пределах TTL хранения
  (frozen — более долгий класс). **Outbox — временный буфер доставки** и *может* потерять сообщение
  до того, как офлайн-получатель получит его вживую.
- History и Outbox пишутся **в одной транзакции**, поэтому всё, что в Outbox, уже есть в History
  (`persist → видно через REST → пригодно для GC`, именно в этом порядке).
- **GC удаляет `outbox WHERE id < safe`**, где `safe` — наименьший *неподтверждённый* id; сам pending
  никогда не удаляется. Граница двигается по **ACK получателя** *или* по **отключению** (pending
  офлайн-клиента становится собираемым — данные остаются в History).
- Клиент считается **офлайн** для GC по закрытию соединения / ACK; presence (Redis) гейтит только
  *попытки живой доставки*. Сетевой сбой может дать кратковременный ложный offline — это безопасно:
  данные в History, клиент догоняется через REST при reconnect.
- **При (пере)подключении клиент вычитывает историю через REST, затем продолжает по WS.** Потерь
  между ними нет; возможны дубли (REST-страница + live) → **клиент дедуплицирует по `messageId`**.

## 🧱 Архитектура — гексагональное ядро + реактивный конвейер

```
domain/   — модели без фреймворков (Message, Conversation, ClientData, …) и контракты
core/     — бизнес-логика: стадии конвейера, резолвер, поллеры, watermark, presence, GC
ports/    — driving/driven интерфейсы (history, outbox, channel, presence, tetris, message, …)
infrastructure/ — адаптеры: Postgres, Redis, WebSocket, REST
```

**Конвейер** — это машина состояний, собираемая под тип сообщения резолвером
(`EnumMap<MessageType, PipelineType>`):

1. **Валидация** — целостность, членство, доступ.
2. **Сохранение** — History + Outbox (стратегия A), через порт.
3. **ACK** — отправителю.
4. **Кэширование** — Redis для идемпотентности/быстрого доступа (кэширующие стратегии).
5. **Доставка** — по WebSocket; помечает `DELIVERED` при успешной отправке.
6. **Финализация** — обработка ошибок / компенсация.

**HexagonalArchitectureTest** (ArchUnit) фиксирует слоистость на этапе сборки: домен остаётся без
фреймворков, ядро зависит только от портов, а порты не зависят от ядра или инфраструктуры.

### Реактивная модель и управление нагрузкой

Стадии работают асинхронно с реактивными буферами между ними; при перегрузке чтение из outbox
замедляется (адаптивная обратная связь `Regulator`) и автоматически восстанавливается. Все метрики
пропускной способности/задержек/буферов экспортируются в **Prometheus** (`/q/metrics`).

## 🗄️ Движок доставки — Outbox + History + Watermark/Tetris GC

- **Собственный PostgreSQL-outbox.** Входящие persistent-сообщения попадают в `outbox`; поллер
  читает напрямую из Postgres (**Kafka во внутреннем движке нет**), доставляет, строки чистятся по
  подтверждению.
- **History** — надёжный источник истины; **Outbox** — буфер живой доставки.
- **Лизинг.** Поллер забирает батч **одним оператором** —
  `UPDATE outbox … FROM (SELECT id … WHERE lease_until <= now() ORDER BY id LIMIT n FOR UPDATE SKIP LOCKED) … RETURNING …` —
  поэтому несколько инстансов никогда не обрабатывают одну строку, и за опрос — один round-trip.
- **GC по watermark'у «Tetris».** Диапазоны ACK по получателям сливаются в **Redis** (ZSET + атомарные
  **Lua**-скрипты) для вычисления **глобального безопасного id удаления**; GC удаляет `outbox WHERE
  id < safe`. History сохраняется по TTL. Скрипты вызываются через `EVALSHA` с **фоллбэком на `NOSCRIPT`**
  (переживает рестарт Redis); стартовая очистка использует `SCAN` + `UNLINK` (а не блокирующий `KEYS`).
  См. [Приложение A](#-приложение-a--модель-tetris-ackgc).
- **Redis — перестраиваемый кэш; источник истины — Outbox.** Состояние Tetris можно потерять (flush /
  холодный старт) без потери данных: GC **гейтится флагом `tetris:primed`** и при «непраймленном»
  состоянии **регидрирует pending из Outbox** (`recipientId → id строк`) перед вычислением `safe`.
  Поэтому GC не может уйти вперёд и преждевременно подрезать недоставленные строки, забытые кэшем.
  (Redis-кластер делает потерю редкой; регидрация делает её безопасной в любом случае.)

## 🟢 Связка Presence ↔ GC

Presence — это **зависимость корректности GC**: watermark продвигается при доставке, что требует,
чтобы получатель был *достоверно* онлайн — фантомно-онлайн клиент останавливает GC. Поэтому:

- **Presence (Redis)** гейтит только *попытки живой доставки* (офлайн → держим в Outbox); это **не**
  источник истины для GC. Watermark GC двигается по **ACK получателя** или по **закрытию соединения**.
- **Пометка OFFLINE при сбое доставки** — если живая отправка в открытую сессию падает, получатель
  переводится в OFFLINE, чтобы поллер прекратил фантомно-онлайн ретраи и просто держал сообщение в
  Outbox.
- **Heartbeat / ping → reaper.** Сервер авто-пингует; `pong` живого клиента обновляет его активность
  (`@OnPongMessage`). Соединение, переставшее отвечать дольше `processing.session.idle-timeout-ms`
  (по умолчанию 35 c), **принудительно закрывается** reaper'ом → `@OnClose` → очистка presence/watermark.
  Это держит presence честным и исключает остановку GC из-за фантомно-онлайн клиентов.

## 🛡️ Идемпотентность

- **Сервер:** best-effort дедуп через буфер с коротким TTL по ключу `messageId` (Redis), рассчитанный
  на сетевые задержки + ACK-flush + интервал планировщика.
- **Клиент:** авторитетный дедуп по `messageId` — именно поэтому exactly-once не является целью
  сервера. Редкие дубликаты на проводе допустимы; пользователь их не видит.

## 🔌 Переподключение и синхронизация истории

При (пере)подключении клиент **вычитывает историю своего разговора по REST**, затем возобновляет
живую доставку по WS — именно это сквозное чтение гарантирует, что ничего не потеряно между
отключениями.

Синхронизация истории **ограничена разговором и постранична по курсору**:
`GET /api/chats/{id}/messages?since=<последний messageId>&limit=<n>` (по умолчанию 200, максимум 500).
Поскольку `messageId` — это ULID, он же служит курсором страницы
(`WHERE message_id > $since ORDER BY message_id`).

## 🔐 Аутентификация и авторизация

- **Ory**-идентичность, инжектируемая на границе (Oathkeeper) в виде заголовков — сервис им доверяет,
  он **не** валидирует JWT сам:

  | Заголовок | Значение |
  |-----------|----------|
  | `X-User-Id` | идентичность вызывающего (обязательно) |
  | `X-User` | отображаемое имя |
  | `X-Role` | `MASTER` / `CLIENT` |
  | `X-User-Email` | email |

  Отсутствует/пустая идентичность → закрытие WS / HTTP `401`.
- **Авторизация по членству в разговоре** — читать/писать в чат могут только два его участника;
  остальные получают `403`. Пары из чёрного списка не могут переписываться.

## 🌐 HTTP- и WebSocket-API

### WebSocket-протокол

Подключение к `ws://<host>/ws` с **Ory-заголовками на handshake** (как в REST). Сервер доверяет
`X-User-Id` сессии как аутентифицированному отправителю. Кадры — JSON-объекты `Message`.

**Схема кадра** (набор полей зависит от `type`):

| поле | кто ставит | смысл |
|------|------------|-------|
| `type` | оба | `MessageType` (таблицы ниже) |
| `senderId` / `recipientId` | клиент (in) / сервер (out) | идентичности |
| `conversationId` | оба | id чата (пара `(client,master)`); `system:<id>` для системных нотисов |
| `messageId` | клиент при отправке → **сервер переназначает ULID** | серверный id монотонный + курсор истории |
| `correlationId` | оба | связь ACK/квитанции с `messageId` |
| `ackId` | клиент в `*_ACK` | id outbox-строки, которую квитируют (двигает GC-watermark) |
| `payload.kind` / `payload.body` | оба | `text` · `custom` · `event` · `attachment`; `body` = контент/ссылка |
| `meta` | оба | непрозрачная map (напр. `orderId`, `objectKey/fileName` вложения) |
| `senderTimestamp` / `senderTimezone` | клиент | epoch ms + IANA tz (валидируется, ≤ 5 мин скью) |
| `serverTimestamp` / `id` / `sequenceNumber` | сервер | назначаются сервером |

**Входящие — клиент → сервер** (только эти принимаются):

| `type` | назначение |
|--------|------------|
| `CHAT_IN` | отправка чат-сообщения (persistent, Strategy A) — проверка членства + blacklist |
| `SIGNAL_IN` | WebRTC offer/answer/ICE (Strategy S, эфемерно) — та же проверка членства |
| `CHAT_ACK` | delivery-ACK получателя: `correlationId`=доставленный `messageId`, `ackId`=outbox id → `SENT→DELIVERED` |
| `READ_IN` | получатель прочитал разговор → `READ` + push `READ_OUT` отправителю |
| `SYSTEM_ACK` | подтверждение системного нотиса: `correlationId`=`messageId` нотиса → ретракт dead-letter-драфта |

**Исходящие — сервер → клиент:**

| `type` | назначение |
|--------|------------|
| `CHAT_OUT` | доставленное чат-сообщение |
| `CHAT_ACK` | квитанция `SENT` отправителю (`correlationId` = отправленный `messageId`) |
| `SIGNAL_OUT` | доставленный signaling-кадр |
| `READ_OUT` | «твои сообщения прочитаны» → исходному отправителю |
| `PRESENT_INIT` / `PRESENT_JOIN` / `PRESENT_LEAVE` | снимок presence / пир онлайн / пир офлайн |
| `SYSTEM_OUT` | must-arrive системный нотис (Strategy C) — подтверждается `SYSTEM_ACK` |
| `SERVICE_OUT` | транзиентный сервисный нотис (напр. перегрузка/backpressure на входе) |

Доставка **at-least-once**; клиент **дедуплицирует по `messageId`** и **пересортирует по монотонному
id** (порядок — клиентский). Пустая/отсутствующая identity на handshake → сокет закрывается.

### REST — `/api/chats`

| Метод и путь | Назначение | Заметные ответы |
|--------------|------------|-----------------|
| `POST /api/chats` | Создать/вернуть чат по `{clientId, masterId, metadata}` (идемпотентно) | `201` создан / `200` существует |

### Kafka — `order.events` (входящее)

Сервис **потребляет** `order.events` (канал `order-events-in`, SmallRye Kafka) — второй driving-адаптер
над core-операциями создания чата / заморозки (концепт §2, ADR-007). JSON-конверт:

```jsonc
{
  "eventId":    "uuid",                 // ключ дедупа продюсера (at-least-once; консьюмер идемпотентен)
  "eventType":  "order.master.interested",
  "occurredAt": "2026-06-24T10:00:00Z",
  "payload": { "clientId": "...", "masterId": "...", "orderId": "..." }  // Ory id + непрозрачный orderId
}
```

| `eventType` (через конфиг) | действие |
|----------------------------|----------|
| `order.master.interested` (`messenger.order-events.type.master-interested`) | **UC-H01** создать чат `(client,master)` (идемпотентно, `orderId` в metadata) |
| `order.contract.concluded` (`messenger.order-events.type.contract-reached`) | **UC-H04** найти чат по паре → заморозить сообщения этого заказа |

> **Контракт предполагаемый — сверить с командой Order.** Конверт по ADR-007; конкретные строки
> `eventType` — конфиг (`messenger.order-events.type.*`), сверка = конфиг, не код.

- ✅ **Strategy C + dead-letter** — must-arrive системные нотисы, eager-draft + retract-on-ACK (ADR-014).

**Частично / отложено:**

- 🟡 **Порядок внутри разговора** (FR-MSG-07) — не реализован (клиент пересортирует по монотонному id; ADR-013).
- ⚪ **Лимитирование по кредитам** (FR-SEC-03) — опционально для v1, отложено.

## 🧩 Приложение A — модель Tetris ACK/GC

Граница безопасного удаления вычисляется слиянием диапазонов ACK по получателям — метафора
**«Tetris»**: каждый ACK заполняет ячейку, заполненные слои можно очистить.

**Реализованный вариант — Redis + Lua (v1).** Структуры Redis:

```
tetris:re:<id>:ack   ZSET   — неподтверждённые id сообщений получателя
tetris:minids        ZSET   — минимальный ожидающий id на клиента (score = id)
tetris:re:cnt        HASH   — число ожидающих на клиента (детект «тяжёлых» клиентов)
tetris:lastid        STRING — последний глобально увиденный id (продвижение вперёд)
```

Атомарные Lua-скрипты обрабатывают `onSent` / `onAck` / `onDisconnect` и
`computeGlobalSafeDeleteId` (наименьший ненулевой score в `tetris:minids`, иначе `lastid + 1`).
Скрипты вызываются через `EVALSHA` с фоллбэком `NOSCRIPT`→`EVAL`.

**Отложенный вариант — Debezium / Kafka-Streams.** Более тяжёлый вариант (Postgres → Debezium →
Kafka, агрегация ACK в Kafka Streams, компактируемые топики `ack_ranges`/`global_offset`)
задокументирован как возможный путь масштабирования в будущем, но **не используется** — внутренний
движок доставки построен только на PG-outbox + Redis.

## 📖 Приложение B — глоссарий

| Термин | Значение |
|--------|----------|
| **Conversation (разговор)** | Пара `(clientId, masterId)`; единица чата. 1:1, без привязки к заказу. |
| **Pipeline (конвейер)** | Последовательность асинхронных стадий по типу: валидация → сохранение → ACK → доставка → финализация. |
| **Outbox** | Буфер живой доставки в Postgres; строки чистятся по подтверждению. |
| **History** | Надёжное долговременное хранилище сообщений (источник истины). |
| **Watermark** | Граница безопасного удаления; строки Outbox ≤ неё собираются GC. |
| **Tetris** | Слияние диапазонов ACK (Redis ZSET + Lua), вычисляющее глобальный безопасный id удаления. |
| **Leasing (лизинг)** | `FOR UPDATE SKIP LOCKED` + лизинг, чтобы инстансы не обрабатывали строки Outbox дважды. |
| **Presence** | Кэшированное в Redis состояние онлайн/офлайн; управляет доставкой и корректностью GC. |
| **Freeze (заморозка)** | Необратимая неизменяемость на уровне сообщения, по заказу (доказательство договора). |
| **ULID** | Монотонный по времени id сообщения; также курсор страницы истории. |
| **Стратегии A/B/C/S** | persistent / fire-and-forget / retry-then-purge / signaling. |
| **Идемпотентность** | Best-effort дедуп на сервере по `messageId`; авторитетен клиент. |
| **Backpressure** | Адаптивное замедление чтения Outbox под нагрузкой (обратная связь Regulator). |
| **Ory (Kratos/Oathkeeper)** | Идентичность; граница инжектирует заголовки `X-User-*`, которым доверяет сервис. |
